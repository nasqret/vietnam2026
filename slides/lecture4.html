Why trust a machine?

A deliberate architectural choice: only a small type-checking kernel must be correct.

Whatever produced a proof, it is ultimately a term; the kernel verifies that term has the claimed type. Elaborator, tactics, pretty-printer — tens of thousands of lines — may all have bugs without endangering soundness.

This is the de Bruijn criterion: "the computer says the proof is correct" means something precise.

Two modes, one term

Every proof is a term — you can write it two ways. Term mode: write the inhabitant directly. Tactic mode: a by block whose tactics build the term.

theorem modus_ponens {P Q : Prop} : (P → Q) → P → Q :=
  fun f p => f p                       -- term mode: modus ponens IS application

theorem modus_ponens' {P Q : Prop} : (P → Q) → P → Q := by
  intro f p
  exact f p                           -- tactic mode: same kernel term

#print modus_ponens' reveals fun f p => f p. They differ only in how much of the term you write.

Insist on the framing: these are not two kinds of proof. A by block is a metaprogram — the tactics build the very same lambda term that term mode writes by hand, and when it finishes, Lean hands the kernel a term either way. Walk modus ponens: in term mode it is literally function application, the fun f p arrow f p they met in Lecture 1. The receipt is #print — run it on the tactic proof and the same term falls out. If asked about universes, keep it small: Prop is Sort 0 for proof-irrelevant propositions, Type is Sort 1 for data.

Symmetry of ∧, both ways

Conjunction is a structure with two fields — a proof of $P \land Q$ is a pair, and swapping is trivial.

-- term mode: destructure the pair, rebuild it swapped
theorem and_comm_term {P Q : Prop} : P ∧ Q → Q ∧ P :=
  fun ⟨hp, hq⟩ => ⟨hq, hp⟩

-- tactic mode: the SAME term, built step by step
theorem and_comm_tac {P Q : Prop} (h : P ∧ Q) : Q ∧ P := by
  obtain ⟨hp, hq⟩ := h    -- split into hp : P, hq : Q
  constructor             -- goal Q ∧ P → two goals: Q, then P
  · exact hq
  · exact hp
Conjunction is a structure with two fields, so a proof of P and Q is honestly a pair — swapping the components is the entire proof. Walk the tactic version line by line: obtain splits the hypothesis into hp and hq, constructor splits the goal Q and P into its two fields, and the two exacts fill them in. Then deliver the receipt: #print and_comm_tac shows fun h maps to the pair of h.2 and h.1 — the anonymous constructor and the projections, the same kernel term. Term mode and tactic mode differ only in how much of the term you write yourself.

Propositions vs data

A P : Prop is proof-irrelevant: any two proofs of P are equal. Data n : Nat is not — 2 and 3 differ. So you may not pattern-match a proof to extract data (large elimination).

The Curry–Howard dictionary as concrete datatypes:

  • $P \Rightarrow Q$ — a function P → Q
  • $P \land Q$ — a pair / structure And P Q
  • $P \lor Q$ — a sum Or P Q
  • $\exists x,\ p\,x$ — a dependent pair Exists p
  • $\forall x,\ p\,x$ — a dependent function (x : α) → p x

This is the distinction that trips up every mathematician, so say it twice: any two proofs of a proposition are equal — a proof carries no information beyond the fact that it holds — whereas 2 and 3 are genuinely different data. That is why you may not pattern-match on a proof to extract data: if you could pull the witness out of an existential, two equal proofs would map to unequal numbers, a contradiction. Then walk the dictionary and stress it is Lecture 3's table, now as concrete datatypes. An existential is a pair

The punchline students remember: $\exists$ is not a mystical quantifier — its constructor is $\langle \text{witness},\ \text{proof} \rangle$.

-- the proof IS the pair (witness 2, a proof that 2^2 = 4)
theorem exists_sq_four : ∃ n : Nat, n ^ 2 = 4 := ⟨2, rfl⟩

rfl works because $2^2$ computes to $4$: both sides share a normal form, so they are definitionally equal. Likewise 2 ∣ n is by definition ∃ c, n = 2 * c — the same proposition.

equiv POW 2 2 = 4

This is the punchline students remember, so give it room: the existential is not a mystical quantifier, it is a pair type — witness in one slot, proof in the other, and its eliminator is just destructuring. Ask the room why rfl proves two squared equals four before revealing it: both sides compute to the same normal form, and that is what definitional equality means. Then land the slogan they will meet constantly: two-divides-n IS exists c, n equals 2c, by definition — no lemma needed to pass between them. The equiv command in the lab is exactly this check, run in pure lambda calculus: normalize both sides and compare.

The naturals and induction

Nat is an inductive type — Peano's axioms as constructors. Addition recurses on its second argument:

$$ m + 0 = m, \qquad m + (n+1) = (m + n) + 1. $$

This asymmetry is the surprise of the day: $m + 0 = m$ is a defining equation (holds by rfl), but $0 + n = n$ is not — the recursion never inspects a left zero, so it must be proved by induction.

Then sell the asymmetry as the single most instructive surprise of the day: addition recurses on its second argument, so m plus zero equals m is a defining equation, but zero plus n never fires the recursion. Before revealing which is which, ask the room: which of the two do you expect to hold by pure computation? The wrong guesses are the lesson.

Commutativity, in four lines

On a private copy of the naturals, rebuild it from the two defining equations (as in NNG4):

theorem zero_add (n : N) : add .zero n = n := by
  induction n with
  | zero      => rfl
  | succ k ih => rw [add_succ, ih]

theorem add_comm (m n : N) : add m n = add n m := by
  induction n with
  | zero      => rw [add_zero, zero_add]
  | succ k ih => rw [add_succ, succ_add, ih]

The successor case assumes ih : add m k = add k m and rewrites the goal to a syntactic identity. Exactly NNG4's Addition World star.

Read add_comm slowly, because it is the whole method in four lines. Induction on n splits into the base case, closed by rewriting both sides down to m, and the successor case, which assumes the induction hypothesis ih and rewrites with add_succ, then succ_add, then ih, until the goal is a syntactic identity. Note we needed zero_add and succ_add proved first — each itself by induction — so lemmas stack. Tell them this is precisely the milestone the Natural Number Game's Addition World hands out a star for, and that NNG4 in the browser is the homework ramp.

Rewriting: rfl, rw, simp

  • rfl — closes a = a when both sides are definitionally equal. Proves $m+0=m$ and $2^2=4$; not $0+n=n$.
  • rw [h] — rewrites the goal left-to-right by h. Fails if the LHS is not present syntactically. Use rw [← h] for the reverse; chain rw [h₁, h₂].
  • simp — repeatedly rewrites with the whole @[simp] set until nothing applies. Powerful but opaque — prefer targeted rw for a known step.
Three tools, three sharply different jobs — make each contrast concrete. rfl closes only definitional equalities, where both sides reduce to a common normal form: it proves m plus zero equals m and two squared equals four, but not zero plus n. rw is strictly left-to-right and fails when the left-hand side is not present syntactically — drill the reverse form rw with the back-arrow, because forgetting it is the usual beginner friction. simp is powerful but opaque: the working advice is a targeted rw when you know the exact step, and simp only with a named lemma list when you want control.

A first honest end-to-end proof

Gauss's sum, kernel-checked: $$ 2\sum_{k=0}^{n} k = n(n+1). $$

import Mathlib

theorem gauss (n : Nat) :
    2 * (∑ k ∈ Finset.range (n + 1), k) = n * (n + 1) := by
  induction n with
  | zero      => simp
  | succ n ih => rw [Finset.sum_range_succ, Nat.mul_add, ih]; ring

The step peels off the last summand, distributes, substitutes ih, and lets ring clear the polynomial dust — a complete sorry-free proof.

This is the first honest end-to-end theorem of the course, so savor it — a real identity, kernel-checked. The base case is simp: it knows the empty sum reduces and zero equals zero times one. The successor step is the whole idiom in one line: Finset.sum_range_succ peels off the last summand, Nat.mul_add distributes, ih substitutes the induction hypothesis, and ring clears the polynomial dust. Emphasize there is no sorry anywhere — and that #print axioms gauss is the audit that certifies it.

…and cite the library

The other mental model: the right move is usually to find the lemma. Euclid's theorem is already Nat.exists_infinite_primes.

theorem prime_above (N : Nat) : ∃ p, N < p ∧ p.Prime := by
  obtain ⟨p, hN, hp⟩ := Nat.exists_infinite_primes (N + 1)  -- gives N+1 ≤ p
  exact ⟨p, by omega, hp⟩                                    -- omega: N+1 ≤ p ⟹ N < p

Find the name, destructure its output, let omega finish. Searching (exact?, Loogle, LeanSearch) is a core skill, not cheating.

Now the other mental model: the right move is usually to find the lemma, and both models must coexist. Euclid's theorem is already in Mathlib as Nat.exists_infinite_primes — we did not reprove it; we cited it at N plus one, destructured the triple it returns, and let omega promote N plus one at most p into N strictly below p. Say it explicitly to the room: searching with exact?, apply?, Loogle, and LeanSearch is a core skill, not cheating — the same idiom makes irrationality of root two a one-liner in the next lecture.

Automation: match the shape

The cure for love/burn with automation is knowing each tactic's scope.

  • decide — a decidable prop, by evaluating its instance. Fails if undecidable or too large.
  • omega — linear integer/nat arithmetic (Presburger). Will not touch any product of variables $x \cdot y$.
  • ring — identities in a commutative (semi)ring. Not inequalities.
  • linarith / nlinarith — inequalities from hypotheses; needs the right hints.

"Just try omega" on a nonlinear goal is a real time sink.

Beginners either fall in love with automation or get burned by it, and the cure for both is knowing each tactic's scope. Go down the list: decide evaluates a Decidable instance, so it needs decidability and can hang on large instances; omega is Presburger arithmetic — linear only, it will never touch a product of variables; ring proves commutative-ring identities but no inequalities; linarith wants the right hypotheses as hints. Kill the misconception by name: just-try-omega on a nonlinear goal is a real time sink. The mantra to leave them with: match the tactic to the shape of the goal.
